## 1. Real Token Counting

### Current state
`base.ts:countTokens()` estimates via `Math.ceil(chars / 3.5)` — off by up to 30% depending on language and code density.

### Why it's easy
`tiktoken` is already in `package.json` dependencies. One function change.

### Implementation

**`src/providers/base.ts`**
```typescript
import { get_encoding } from 'tiktoken'

// Lazy-init: encoding loads WASM once, reuse across calls
let enc: ReturnType<typeof get_encoding> | null = null
function getEnc() {
  if (!enc) enc = get_encoding('cl100k_base')
  return enc
}

// In BaseProvider:
countTokens(messages: Message[]): number {
  const enc = getEnc()
  return messages.reduce((sum, m) => sum + enc.encode(m.content).length, 0)
}
```

**Notes:**
- `cl100k_base` is GPT-4's tokenizer. Both GLM and DeepSeek use similar BPE tokenization — margin of error drops from ~30% to ~5%.
- `tiktoken` uses WASM internally; lazy-init avoids startup cost.
- No new dependencies needed.

## 3. Permission System

### Current state
- `PermissionMode` type exists (`ask | auto | deny`)
- `allowedCommands` and `allowedWriteDirs` in config
- **Not enforced**: `loop.ts` executes all tools unconditionally regardless of `permissionMode`

### Design

Three enforcement levels (matching Claude Code):
- `auto` — execute without prompting
- `ask` — prompt before each tool call, user approves/denies
- `deny` — block all tool execution

Per-tool config (new in `AudreyConfig`):
```typescript
toolPermissions?: Record<string, 'allow' | 'ask' | 'deny'>
// e.g. { bash: 'ask', read_file: 'allow', write_file: 'ask', fetch: 'ask' }
```

Default permissions by tool risk level:
| Tool | Default |
|------|---------|
| `read_file`, `grep`, `glob` | `allow` |
| `write_file`, `bash` | `ask` in `ask` mode, `allow` in `auto` mode |
| `fetch`, `search` | `ask` |

### Implementation

**`src/agent/loop.ts`** — add permission check before tool execution:
```typescript
// Before: const result = tool ? await tool.execute(...) : ...
// After:
const permitted = await checkPermission(call.name, call.args, config, permissionMode, askUser)
if (!permitted) {
  yield { type: 'tool_result', id: call.id, name: call.name, result: '[denied by user]', isError: true }
  continue
}
```

**`src/tools/permissions.ts`** (new file):
```typescript
export async function checkPermission(
  toolName: string,
  args: Record<string, unknown>,
  config: AudreyConfig,
  mode: PermissionMode,
  askUser: (msg: string) => Promise<boolean>,
): Promise<boolean> {
  if (mode === 'auto') return true
  if (mode === 'deny') return false
  // mode === 'ask': check per-tool override, then prompt
  const override = config.toolPermissions?.[toolName]
  if (override === 'allow') return true
  if (override === 'deny') return false
  return askUser(`Allow ${toolName}(${JSON.stringify(args).slice(0, 80)})?`)
}
```

**UI: permission prompt** — `agentLoop` needs a callback from App.tsx:
```typescript
// agentLoop signature change:
export async function* agentLoop(
  messages: Message[],
  provider: IProvider,
  config: AudreyConfig,
  opts?: { maxIterations?: number; onPermissionRequest?: (msg: string) => Promise<boolean> }
): AsyncGenerator<LoopEvent>
```

The `onPermissionRequest` callback renders an inline `[y/N]` prompt in the TUI.

**Parallel tool calls** — currently tools execute sequentially. With permission checks, parallel is natural:
```typescript
// Current: for (const call of toolCalls) { await ... }
// v0.1.1: Promise.all for tools that don't write to same path
const results = await Promise.all(toolCalls.map(call => executeWithPermission(call)))
```

**Timeout + retry** — already in `BaseProvider.chat()` for HTTP. Add to tool execution:
```typescript
const result = await Promise.race([
  tool.execute(call.args as any, config),
  new Promise<never>((_, reject) =>
    setTimeout(() => reject(new Error(`Tool ${call.name} timed out`)), config.bashTimeoutMs)
  )
])
```

## 4. MCP Extension

### Current state
- `McpServerConfig` type defined in `types.ts`
- `mcpServers` config exists with 3 preconfigured servers
- `@modelcontextprotocol/sdk` in `package.json`
- No actual MCP client code exists

### Design

MCP lets external processes expose tools to Audrey. The flow:
```
audrey startup
  → start configured MCP servers (stdio child processes)
  → discover their tools via MCP protocol
  → register tools into ALL_TOOLS
  → AI can call them like built-in tools
```

#### `src/mcp/client.ts` (new)
```typescript
import { Client } from '@modelcontextprotocol/sdk/client/index.js'
import { StdioClientTransport } from '@modelcontextprotocol/sdk/client/stdio.js'

export class McpClient {
  private client: Client
  private transport: StdioClientTransport

  async connect(name: string, cfg: McpServerConfig): Promise<void>
  async listTools(): Promise<ToolSpec[]>
  async callTool(name: string, args: Record<string, unknown>): Promise<string>
  async disconnect(): Promise<void>
}
```

#### `src/mcp/registry.ts` (new)
```typescript
// Manages all MCP server connections
export class McpRegistry {
  private clients = new Map<string, McpClient>()

  async startAll(mcpServers: Record<string, McpServerConfig>): Promise<void>
  async getAllTools(): Promise<Tool[]>  // returns Tool[] compatible with ALL_TOOLS
  async stopAll(): Promise<void>
}
```

#### Integration points
1. **Startup** (`src/cli.tsx`): `await mcpRegistry.startAll(cfg.mcpServers)` before render
2. **Tool list** (`src/tools/index.ts`): `export const ALL_TOOLS = [...BUILTIN_TOOLS, ...mcpRegistry.tools]`
3. **Shutdown**: MCP processes need cleanup on Ctrl+C

#### Preconfigured servers (already in `DEFAULT_CONFIG`)
- `@modelcontextprotocol/server-filesystem` — file ops (wraps read/write/glob)
- `@modelcontextprotocol/server-fetch` — HTTP fetch (this gives network access for free if MCP ships first)
- `@modelcontextprotocol/server-sequential-thinking` — structured reasoning

#### Caveat
MCP servers are separate Node.js processes. Each `npx -y` install takes 2-10s on first run. Consider caching or pre-installing.
